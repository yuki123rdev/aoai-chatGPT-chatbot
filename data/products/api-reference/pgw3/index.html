Payment Gateway (1.0.0)

Payment Gateway as one of the Rakuten Payment Platform products enables you to use various payment features via our APIs.

# API type Details
1 /Payment/V1/Authorize We provide the following types of APIs for payment processing such as payment authorization and billing.
The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.
  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".
2 /Payment/V1/Authorize/HTML We provide the following types of APIs for payment processing such as payment authorization and billing.
The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.

  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".
3 /Payment/V2/Authorize/HTML We provide the following types of APIs for payment processing such as payment authorization and billing.
The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.

  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".
4 /Payment/V1/AuthorizeAndCapture We provide the following types of APIs for payment processing such as payment authorization and billing.
The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the payment status, if you use AuthorizeAndCapture, the status will be "captured", skipping the "authorized" status. See the detail of the payment status transition - 4. Functional Document#4.FunctionalDocument-PaymentStatusType

  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".
5 /Payment/V1/AuthorizeAndCapture/HTML We provide the following types of APIs for payment processing such as payment authorization and billing.
The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the payment status, if you use AuthorizeAndCapture, the status will be "captured", skipping the "authorized" status. See the detail of the payment status transition - 4. Functional Document#4.FunctionalDocument-PaymentStatusType

  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.

  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".
6 /Payment/V2/AuthorizeAndCapture/HTML We provide the following types of APIs for payment processing such as payment authorization and billing.
The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the payment status, if you use AuthorizeAndCapture, the status will be "captured", skipping the "authorized" status. See the detail of the payment status transition - 4. Functional Document#4.FunctionalDocument-PaymentStatusType

  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.

  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".
7 /Payment/V1/Modify This API will allow modifying (increase or decrease) the amount which is already "authorized" or "captured" and granted Rakuten Points. You only need to send both the modified amount and the granted Rakuten Points, not the amount of the redeemed Rakuten Points. This API is not provided for non-changeable payment methods such as cash payment. If the Modify API request is processed successfully, the amount will be changed, and the transaction will keep the previous payment status type.
8 /Payment/V1/Capture The Capture API processes billing for "authorized" status transactions. The authorization process must be first if you want to separate your payment process into authorization and capture for billing.
9 /Payment/V1/Find The Find API allows getting a certain transaction registered in the PGW system, such as the current payment status of the transaction. The Find API can also be used to verify the status of a certain payment transaction when no response was provided after making a payment request.
10 /Config/V3/Set This API provides the function to create a merchant.


A merchant is not activated immediately just after you call this API and its timing depends on each payment agency.
11 /Config/V3/Get This API provides the function to get merchant information.

If you have anything to share to user except Rrerequite, please write it down here.
12 3DSecureEnrollment/V2/Check “Please refer to the implementation guide of 3D Secure 2.0 (EMV 3D-Secure) to implement this API “3DSecureEnrollment/V2/Check.”
13 /Payment/V1/CancelOrRefund This API works for cancellation or issuing a refund, depending on the payment status. If the payment status is not captured yet and is still in authorized status, it works to cancel the authorized transaction. If the payment status is already captured, it works to issue a refund.

We add new parameters without notice. So please don't have a check to reject the response with a parameter you don't recognize.
We provide the following types of APIs for payment processing such as payment authorization and billing.

The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the payment status, if you use AuthorizeAndCapture, the status will be "captured", skipping the "authorized" status. See the detail of the payment status transition PaymentStatusType
  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.
  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".

V1/AuthorizeAndCapture/HTML

See the differences among other APIs
We add new parameters without notice. So please don't have a check to reject the response with a parameter you don't recognize.
We provide the following types of APIs for payment processing such as payment authorization and billing.

The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the payment status, if you use AuthorizeAndCapture, the status will be "captured", skipping the "authorized" status. See the detail of the payment status transition PaymentStatusType
  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.
  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".

V2/AuthorizeAndCapture/HTML

See the differences among other APIs
We add new parameters without notice. So please don't have a check to reject the response with a parameter you don't recognize.

We provide the following types of APIs for payment processing such as payment authorization and billing.

The main differences are how the payment status changes and how the end user's browser is redirected. The interfaces are almost identical, and the request and response formats are the same.

  • As for the payment status, if you use AuthorizeAndCapture, the status will be "captured", skipping the "authorized" status. See the detail of the payment status transition PaymentStatusType
  • As for the redirect, you will need to use the HTML version, not the API version, if your payment method requires the shopper's browser to be redirected or enter anything, such as bank payment or 3D Secure for credit card payments.
  • When you use Authorize type API, you will also need to use /Payment/V1/Capture. By calling this API, the payment status will be changed from "authorized" to "captured".

V1/Modify

This API will allow modifying (increase or decrease) the amount which is already "authorized" or "captured" and granted Rakuten Points. You only need to send both the modified amount and the granted Rakuten Points, not the amount of the redeemed Rakuten Points. This API is not provided for non-changeable payment methods such as cash payment. If the Modify API request is processed successfully, the amount will be changed, and the transaction will keep the previous payment status type.

How Modify API works
  • The Modify API internally creates a new transaction with a new amount first and cancels the old one after the new transaction is succeeded. This process creates another authentication/capture before we cancel/refund the original transaction. Therefore, a credit from the shopper's credit card account is additionally needed for the new amount transaction. You might get an error due to a lack of credit limit following the shopper's credit card account.


Remarks on credit card payments
  • Because of this behaviour for payment processing, in the case of credit card payment, the liability shift on 3D Secure is reset even if the original transaction went through 3D Secure authentication properly. Only Rakuten Card and E-Scott support a partial refund, which doesn't force us to create a new transaction to have the modified amount transaction. Please confirm the detail with the PGW team when you implement Modify API.
  • It may not be compatible with bonus payments when you make an increased amount of the new transaction, typically after the bonus payment starts.
  • Please confirm the cancel/refund acceptable period of your payment service provider from this page Agency.


Rakuten Coupon
  • The Modify API doesn't support a transaction created with Rakuten Coupon.

Please check out more details on our FAQ page
