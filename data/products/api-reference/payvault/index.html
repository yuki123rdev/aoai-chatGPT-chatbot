Credit Card

Card/V1/Add

Registers a new card. The response contains a cardToken that can be used to charge the corresponding card. It also contains non-sensitive card information: the first 6 digits of the card number (the card IIN), the last 4 digits, and the expiration date.

The request is a JSON document sent via HTTPS. No signature is expected.

In order to restrict your PCI-DSS scope, you must not call this operation directly from your server (that is, from service's or Global Id's servers) without consulting PayVault team in advance. Instead, this operation should be called directly by the shopper's browser. That way, the full card number never passes through your server.

The card token issued by this operation can be used to request a credit card payment authorization via Payment Gateway's Payment API. Note that the service cannot request the full, unmasked card number using the card token.

The card token is guaranteed to be unique: even if /api/pv/Card/V1/Add is called twice with the same parameters, the returned card token will be different each time. Hence, it is strongly recommended that users of this API avoid storing duplicate card token's.

If the card brand cannot be determined from the given card number, then no card token is generated, and the resultType is "failure" with errorCode = "invalid_request_parameter".

Please register test card numbers only when using the staging environment. Refer to /api/pv/Card/V1/Add for a (non-exhaustive) list of test card numbers

Expired Cards

Credit cards will be added regardless of the expiration date due to various security concerns as well as to allow existing services to migrate to PayVault system. However, only valid cards (i.e. not expired yet) can be added when using payvault.js.


Please check out more details on our FAQ page

Card/V1/AddPlus

Registers a new card in PayVault.

The response contains a cardToken that can be used to charge the corresponding card and the card fingerprint. Additionally contains non-sensitive card information: the first 6 digits of the card number (the card IIN), the last 4 digits, and the expiration date.

As opposed to the /api/pv/Card/V1/Add, the request is a signed JSON document sent via HTTPS.


Please check out more details on our FAQ page

Intended usage

The purpose of this API is to reduce redundant calls by internal Rakuten services that require card fingerprint information to perform various operations such as fraud check. The response will be identical with /Add API with the exception of the fingerprint field.

The CVC or CVV (3-4 digit security code) cannot be stored according to PCI DSS requirements and as such is not among the request parameters.

Access to this API is decided on a case by case basis and requires a compelling business requirement for using card fingerprint as it is considered sensitive information.

Card/V1/ForwardResponse

This API is used to replace card token of request data from services with raw card number. It also sends the request data to agency instead of services and then returns response of agency without changing anything.
All forward URLs are defined in the PayVault system and mapped by URL ID of request.
This is a common interface used for all services and agencies and does not return error details because interface is different in each service and agency.


  1. Caller service calls payvault
  2. PayVault forwards request to forwarded service
  3. Forwarded service return response with body and headers to return to caller service, end eventually a token to substitute with card number
  4. PayVault return response with card number
  5. (Optional, in case of error) PayVault notify Forwarded service of error

Card/V2/Add

Registers a new card. The response contains a cardToken that can be used to charge the corresponding card. It also contains non-sensitive card information: the first 6 digits of the card number (the card IIN), the last 4 digits, and the expiration date.
The request is a JSON document sent via HTTPS. No signature is expected.


In order to restrict your PCI-DSS scope, you must not call this operation directly from your server (that is, from service's or Global Id's servers) without consulting PayVault team in advance. Instead, this operation should be called directly by the shopper's browser. That way, the full card number never passes through your server.

The CVC or CVV (3- or 4-digit security code) cannot be stored according to PCI-DSS requirements, so it is not one of the request parameters.

The card token issued by this operation can be used to request a credit card payment authorization via Payment Gateway's Payment API. Note that the service cannot request the full, unmasked card number using the card token.

The card token is guaranteed to be unique: even if /api/pv/Card/V2/Add is called twice with the same parameters, the returned card token will be different each time. Hence, it is strongly recommended that users of this API avoid storing duplicate card token's.

If the card brand cannot be determined from the given card number, then no card token is generated, and the resultType is "failure" with errorCode = "invalid_request_parameter".

Please register test card numbers only when using the staging environment. Refer to /api/pv/Card/V2/Add for a (non-exhaustive) list of test card numbers

Expired Cards
Credit cards will be added regardless of the expiration date due to various security concerns as well as to allow existing services to migrate to PayVault system. However, only valid cards (i.e. not expired yet) can be added when using payvault.js.

This API is used to retrieve full card number and expiration date. So which means that the system calling this API needs to be PCI DSS compliant.

This API requires the caller IP address to be in a IP whitelist. Please share the IPs that will be used in STG and PRO when creating service ID, or through here.
