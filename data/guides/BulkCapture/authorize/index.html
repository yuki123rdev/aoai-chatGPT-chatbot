Overview​

We support a feature to capture multiple transactions at once via file.

You upload a file and we return a response file after capture.

Please note that the feature is for services in Japan.

danger

Files are handled in sequence, not in parallel.

So, we cannot secure when a file is executed clearly.

You can find that by checking transactionTime in each record of output file.

I/F​

Card updater result file​

This file stores the card tokens which were issued by Card Updater. For more information of this file, please see Card Update Guideline#SpecificationofResponsefile.

As of 2021/09/08, the following restrictions apply to the handling of this file.

  • This file is mandatory regardless of whether the parameter withAuthorized at IN file (see "IN file" section below) is true or false.
  • The validity of card tokens in this file is not checked.
  • Only previous month's file is used in the operation. For example if you want to process an IN file named input_20210901.json, the file name of this file must contain 202108 (e.g. res_updateCard_20210801.json).

So the Bulk Capture team plan to implement following features to improve the handling of this file. This will be released in November 2011.

  • Card updater result file is not required if the parameter withAuthorized is true for all records in the IN file.
  • Bulk capture checks validity of the card tokens in card updater result file if if the parameter withAuthorized is false in the IN file.
  • Card update result file can be the current month's file as well as the previous month's file. Current month's file will be used in precedence.

IN file​

File name​

  {service_id}_yyyyMMdd.json

  * If you want to run multiple files on the same day, specify file_id in the file name for each file.
  {service_id}_{file_id}_yyyyMMdd.json

  e.g)
  your-serviceid_first_20221028.json
  your-serviceid_002_20221028.json
  your-serviceid_003_20221028.json

note
  • As of 29 Oct 2018, The date, yyyyMMdd, has no relation when the file is handled. Originally, it was the date when the file is handled, but we remove the spec because there was a requirement to handle a file whose name has old date.
  • As for the date (yyyyMMdd) in request file, we recommend setting the date on which the file is created or uploaded to S3 while you can set your preferred date.
  • Please avoid to set past date (e.g. 20000101) or future date (e.g. 21000101) since the operation will NOT normally proceed.
  • Please also check this note yyyyMMdd

Format​

The IN file must meet all of the following conditions.

  • Maximum number of records in a file is recommended less than 2,000,000.
    • No error would be occurred even it is over 2,000,000, but please make sure to consult with Payment Platform Team in advance if you need to upload a file with over 2,000,000 records.
  • Each request is written per row.
  • UTF-8 must be used as character encoding.
  • No BOM (Bute order mark) is added.
    • For details about BOM, please see https://en.wikipedia.org/wiki/Byte_order_mark.
  • Each row, including the last line, is terminated with UNIX newline character (LF, "\n", 0x0a).
    • Please pay special attention to the last line break, as it is often forgotten.
Sample format of IN file
{"paymentId":"PAYMENTID-0001","serviceID":"sample-service-id",...}
{"paymentId":"PAYMENTID-0002","serviceID":"sample-service-id",...}
{"paymentId":"PAYMENTID-0003","serviceID":"sample-service-id",...}

{"paymentId":"PAYMENTID-XXXX","serviceID":"sample-service-id",...}

Maximum number of records in a file is recommended less than 2.000.000.
No error would be occurred even it is over 2.000.000, but please make sure to consult with Payment Platform Team in advance if you need to upload a file with over 2.000.000 records.

note

In IN file, each request should be written per row. And each row, including the last line should be terminated with UNIX newline character (LF, "\n", 0x0a).

OUT file​

File name​

"_out" will be added to IN file name

e.g)
IN file name        -> OUT file name
{service_id}_yyyyMMdd.json   -> {service_id}_yyyyMMdd_out.json
{service_id}_{file_id}_yyyyMMdd.json  -> {service_id}_{file_id}_yyyyMMdd_out.json

note
  • "yyyyMMdd" of response is the same as that of Request, regardless of the upload or pass date.
    e.g.  Request:  {service_id}_20230101.json → Response: {service_id}_20230101_out.json

FAQ​

QA
1.Does "pending" happen?Yes.

An API call happen internally when the amount is over a threshold defined in advance or you set "true" into withAuthorized.

"pending" happens when a trouble, like a tentative network error, happens on the API request.
2.A request with Rakuten Point without credit card is valid?No.

Credit card is necessary because point may be insufficient when actual logic runs.

In that case, insufficient amount is allocated by credit card.
3.Authentication happens anytime?No.

If amount is 0, it's skipped.
4.Which case is handled by Rakuten Card, not JCN?The following case is handled by Rakuten Card.

・withAuthorized is true
・grossAmount is over 30,000
・A card was not updated by PayVault/JCN, and there is no corresponding record in the file from PayVault
・There is no record in the file for card updater at PayVault whose original token or updated token matches a token you set in the file for bulk capture.

(Because bulk capture has no way to get compnay code, and it's impossible to get the threshold, 30,000, defined based on company code.)
5.Why company code is empty?Your record is handled by JCN or Rakuten Card as above.

- Rakuten Card
The batch calls their API internally.
If it's success, company code never be empty.
If it's failure, there is a possibility that happens.

- JCN>
It depends on the result file of card updater at PayVault.
The file has company code, original token, and updated token.
(Updated token doesn't exist if the card is not updated.)
Card Update Guideline#SpecificationofResponsefile

Bulk capture gets company code from the file from PayVault based on a token you set into the file for bulk capture, bulk capture token.

If there is a record whose original token or updated token matches bulk capture token, the batch retrieves company code from the record.

That means if there is no such record, company code becomes empty.
6.Why Rakuten Point was not redeemed?Even if a shopper has point, it's not redeemed when Rakuten Point has any trouble.

In that case, we just ignore the trouble and handle without Rakuten Point.
7.In case of withAuthorization == true, when is the sale date?In this case, the sale date is set depends on the specifications of Payment Gateway (PGW).

As of August 2021, PGW will set the date when API is called as the sale date.
8.In what cases is the field saleDate ignored even though withAutorization == false?Under the following conditions, Bulk Capture calls PGW API, so the sale date is determined by PGW FAQ No. 7.

As a result, the field saleDate will be ignored.

1.Card amount exceeds floor limit. Note that card amount is described in the field grossAmount at "IN file" section.
2.No valid card token is found, and card amount > 0.
9.In what cases are the field agencyRequestId and reference not be returned even if saleType == saleWithAuhtorize?Since there parameters are set by PGW as a result of an operation between PGW and agency, these may not be returned in some cases.

As of October 2021, the following cases are possible.

・Request validation error
　・Invalid grossAmount is specified (e.g. -1 JPY)
　・grossAmount is larger than the maximum amount
　　・For maximum amount, please see this page: What is the maximum amount for 1 transaction? / 1回の決済にあたり決済金額の上限はいくらですか？
　・Unsupported currency is specified (e.g. USD)
　・Unsupported issuer is specified (e.g. Diners)
・Network error between PGW and agency
