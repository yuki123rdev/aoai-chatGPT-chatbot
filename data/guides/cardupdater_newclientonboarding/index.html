Overvie​

We agree on steps and checkpoints to check when a new client wish to use Card Updater.

Step​

Explanation​

S - card solution team will do

C - cloud team will do

P - payvault team will do

StepTeamPurposeNote
S.1card solution teamcreate business requirement
S.1.1gathering necessary configuration, at lease serviceId, supported card brand & card company
S.2specify S3 folder where client will put request filealso make sure client can connect, upload and download file
C.1cloud teamcreate s3 bucket, grant access permissionshould be on dev, stg & pro environment
C.1.1notify card team once it's donefrom this time, client can send file to s3 folder
S.3card solution teamcreate ticket to payvault team in order to add new client configurationbased on business requirement on step S.1.1.
notify PV up to 2 weeks before client start running on staging (include simulator)
<Necessary info to share with PayVault team via the ticket>
  • subject: register new CU client
  • description: (put link to merchant registration page)
P.1payvault teamset up configurationpayvault needs approvals from both pv team, manager & architect team, might take some times
P.1.1notify card solution team once finish
info

after completing this step, client can do test with simulator on staging by uploading their test file & rule file to designated s3 folder. response file usually is available 5 mins after request is trigger. client can test anytime they want, no need to confirm with payvault.

StepTeamPurposeNote
S.4card solution teamhave credential so that file can be sent to either JCN or RCplease talk with JCN or RC
S.4.1create ticket to payvault and share link to secure box folder, which contains credentialsdouble check content before sharing to payvaul, don't need to wait for response from either payvault or cloud team
P.2payvault teamprovide detail about path and box link above to cloud teampayvault generally can not see the content, so unable to verify.
C.2cloud teamwill upload sftp credential to production server, verify connectionfrom this time, only cloud team can check and verify from production server
S.5card solution teamdecide with client and JCN/RC the time when file will be sentwe call it scheduler on production, please make sure master table page is up to date by this time
S.5.1create ticket to payvault to update schedulerdouble check time in advance
P.3payvault teamupdate scheduler based on provided informationfor sudden change, please talk with PV directly asap.
P.3.1notify card solution team about successful registration of new clientfrom this point, batch will trigger client request file and send to either JCN / RC at designated time based on scheduler
info

after completing this step, client can upload file to their production s3 folder, and expect response at planed time. Consequence run will be as scheduler automatically, no need to confirm with payvault.

Troubleshooting​

Client can not upload file to S3​

→ refer to ticket which cloud team create S3 folder, ask if they granted appropriate permission to target serviceId

Client upload invalid json file to S3​

→ if it's on staging, ask either cloud team or payvault team to remove. If it's on production, create at ticket to cloud team and have manager's approve the operation.

Client upload more than 1 file to s3​

→ for capture request, no problem.

→ for update request, ask either cloud team or payvault team to delete unwanted file. If it's on production, create at ticket to cloud team and have manager's approve the operation.

Client doesn't receive response file​

→ check if they uploaded rule file successful

→ if yes, create a ticket to payvault and ask for investigation, please provide serviceId. PayVault will gather log and share conclusion in ticket. Usually pv will respond in 1 hour.

→ → if system error, payvault will fix

→ → if rule file is invalid, please help client to fix, once finished, share with PayVault.

Response file is unexpected​

→ refer to spec documentation

→ if need to investigate data of card token, ask payvault via a ticket. Usually pv will respond in 1 hour.

Appendix​

Card updater provides 2 features: Capture request and Update request.

Featurerequest S3 folder (where client put file)response folder (where client receive file)use simulator
Capture request(service.id}/request/card-capture/v1/
Request file name must start with (service.id)_
e.g stg-jp-rhikari_request_2021.json		(service.id}/response/card-capture/v1/No
Update request(service.id}/request/card-update/v1/(service.id}/response/card-update/v1/
Response file starts with res_		Yes
