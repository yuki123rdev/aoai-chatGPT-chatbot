1.Purpose of testing using the simulator​

The simulator is used for the purpose of testing between clients (services) and Card Updater on STG environment. It is to simulate the card number update by the card companies instead. Please make sure to test with the simulator before the production release.

<PRO environment>

  • via JCN

  • via Rakuten Card

  • <STG environment(simulator)>

tip

The simulator test does not send data to payment service providers (Rakuten Card PSP, JCN) or credit card companies (Rakuten Card Issuer, Mitsui Sumitomo, etc.), and data is returned by Card Updater.

2.Preparation before using the simulator​

  • Fill out the Merchant Registration Page.(At least 1 week prior to the desired test date)(Necessary items:Service ID, Card Company, Brand)
    By setting up the above information in the Card Updater system, the simulator will be ready to run.

  • Share your preferred dates (term) to conduct simulator test.
    Card Updater needs some setup before conducting the simulator and we also check the availability during the dates which you shared.

  • Access Check to S3 (STG). The simulator uses S3 to send files, so please make sure you can connect to it before testing.
    (For the access point, please refer to Card Update Guideline#FileServer、or "8. S3 access point".
    For details on how to access, please refer to (How do I connect to AWS S3? ) AWS S3への接続方法は？/How to connect to AWS S3? for more information.)

  • Prepare the rule file.
    Only the simulator test requires a rule file. This is to simulate the update of the card number by the credit card companies.
    (or details on how to create a rule file, please refer to "5. How to create a rule file" below.)

  • Prepare the request file.
    Please prepare the card token you want to replace.
    (For the request file, refer to "6. Request file" below, and for how to create the token, refer to "7. How to create the token" below.)

3.Simulator test flow​

  1. Place the rule file in S3.
tip

The card number set in the rule file must be a number that can be successfully checked by Luhn check (check digit).

  1. Place the request file in S3.
tip

Tokenize the credit card number you want to update and set it in the request file.

  1. Card Updater gets the rule file and the request file.
tip

If it was retrieved correctly, the request file will disappear from S3.

  1. Based on the rule file and request file, a response file is created from the request file.

  2. Card Updater put the response file in S3.

  3. Get the response file from S3.

tip

If there are less than 100 recodes, they are usually returned in about 5 minutes after replacement. (This is because the simulator runs every five minutes.)
If the file is not returned, please refer to "9. In case the response file is not returned" below.
Up to 9 million tokens can be tested in a single file, but please contact us in advance if the number exceeds 100,000 tokens.

4.Notes on using the simulator​

  • Once a token has been used by the simulator, the token (card information) cannot be used in the next simulator test.

  • In the next simulator test, please run simulator test with the latest token (card information) that has been renewed.

  • Request file is not deleted by service side. Please ask PGW team to delete the file if necessary.

5.How to create a rule file​

  • Format of the rule file
Data FormatLine Break TimeCharacter EncodingFile Name
AnyLFUTF-8The file name should start with "rule_".
tip
rule_<any string or number>.{any extension}
例 : rule_{YourServiceId}.dat
  • Notes on creating rule files
    • All strings in the rule file must be one-byte characters.
    • Each field in the rule must be separated by a , (comma).
    • Do not include space at the end.
    • Separate records with each rule on a new line.
    • If there are multiple records listed with the same rule, only the rule listed earlier in the file will be applied.
    • The card number set in the rule file must be a number that succeeds by Luhn check (check digit).
    • You must write all rules that apply to the tokens in the request. (See the figure below.)
    • Set this rule at the end of the rule.C. No change to data
ALL,,/*,0,0,0,,

Explanation of how to write the rules​

1. CARD_COMPANY_PATTERN​
#Field NameLengthDescriptionExample Values
1CARD_COMPANY_PATTERN
1 - 15This item is used to set the Merchant ID.

If you have only one Merchant ID, or if you want to target all merchant numbers, please set ALL.

If you have more than one Merchant ID, you can choose which card companies you want to replace.
ALL
38374100001
2CARD_NUMBER_PATTERN1 - 16Specify the card number to be replaced in another request file.

By using wildcards (*), you can specify a range of multiple card numbers.
：Specify all card numbers as targets
41111 * ：Card numbers starting with 41111
 1111：Card numbers ending with 1111
4111 * 1111：Card numbers beginning with 4111 and ending with 1111
3EXP_DATE_PATTERN3 - 5
format MM/YY		Specify the card number to be replaced in another request file by its expiration date.

By using wildcards (*), you can specify a range of multiple months/years.
04/22：April 2022

/ ：All months and any year
12/：December of any years
04/：April of every year
*/22：All months of 2022
4EXPECTED_RESULT_CODE1In case of #1, #2, and #3, specify the result of whether the card is valid or not.0：Success (card number is valid)
1：Failed (card number is invalid)
2：Verification error
5EXPECTED_CARD_CHANGE_SIGN1If the card validity result of #4 is successful, specify whether the card number should be updated as the latest number.

#If #4 is other than success, no setting is required.
0：No update
1：Updated
6EXPECTED_EXP_DATE_CHANGE_SIGN1If the card validity judgment result of 4 is successful,
specify whether or not to update the expiration date as the latest expiration date.

If #4 is other than success, no setting is required.
0：No update
1：Updated
7EXPECTED_NEW_CARD_NUMBER1 - 16If #5 is with card number update, set the new card number to be updated.

If #5 does not update the card number, no setting is required.
4111111111111111
8EXPECTED_NEW_EXP_DATE5
format MM/YY		If #6 is with expiration update, set a new expiration date.

#6 is not required if the expiration date is not renewed.
04/20
12/21

7.How to create a token​

1.Check options (v4. can be selected. Please refer to <image2>.)
2.Check if the status is the same as 1.
3.Press "Load script" button.
4.Check the address. (PayVault API:https://payvault-stg.global.rakuten.com/api/pv/Card/V3/Add
5.Set the "service ID".
6.Set the "False".
7.Set the "card number"
